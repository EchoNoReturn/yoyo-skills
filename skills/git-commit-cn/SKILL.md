# Git 中文提交规范 Skill

使用这个 skill 时，目标是帮助用户生成符合行业规范的中文 git 提交信息，并管理提交和推送流程。

## 工作流程

### 1. 检查 git 状态

首先检查当前 git 仓库状态：

```bash
git status
git diff
git log --oneline -5
```

### 2. 分析变更内容

分析暂存区的变更，识别：
- 新增的文件和功能
- 修改的文件和修复内容
- 删除的文件
- 重构的代码
- 文档更新
- 测试相关变更

### 3. 生成提交信息

根据分析结果，生成符合以下格式的中文提交信息：

```
<类型>(<范围>): <主题>

<body>

<footer>
```

#### 类型说明

- **feat**: 新功能
- **fix**: 修复 Bug
- **docs**: 文档更新
- **style**: 代码格式调整（不影响逻辑）
- **refactor**: 重构（既不是新增功能，也不是修改 Bug）
- **perf**: 性能优化
- **test**: 测试相关
- **chore**: 构建过程或辅助工具的变动
- **revert**: 回滚提交

#### 范围规则

- 使用小写字母
- 使用 kebab-case 格式（如 `user-profile`、`shopping-cart`）
- 范围应反映变更影响的主要模块或组件
- 如果变更影响多个模块，使用主要模块作为范围

#### 主题规则

- 使用中文
- 不超过50个字符
- 使用现在时态（如"添加"而非"添加了"）
- 首字母小写
- 不以句号结尾

#### 正文规则

- 解释为什么修改，而不是怎么修改
- 使用中文
- 每行不超过72个字符

#### 页脚规则

- 关联 Issue 或 PR（如 `Closes #123`、`Fixes #456`）
- 破坏性变更使用 `BREAKING CHANGE:` 开头

### 4. 提交信息示例

```bash
# 正确示例
feat(user): 添加用户登录功能
fix(cart): 修复购物车数量计算错误
docs(readme): 更新项目说明文档
style(components): 统一代码缩进风格
refactor(store): 重构用户状态管理逻辑
perf(image): 优化图片加载性能
test(unit): 添加用户模块单元测试
chore(deps): 升级vue到3.4版本
revert(auth): 回滚用户认证逻辑

# 错误示例
update code  # 缺少类型和中文
修复bug      # 缺少类型和范围
```

### 5. 提交流程

#### 步骤1: 生成提交信息

根据分析结果生成提交信息，格式如下：

```
我为您生成了以下提交信息：

类型: <类型>
范围: <范围>
主题: <主题>

完整提交信息:
<完整提交信息>
```

#### 步骤2: 询问是否提交

```text
是否使用此提交信息进行提交？
1. 是，提交
2. 否，我想修改提交信息
3. 取消提交
```

#### 步骤3: 处理提交

如果用户选择提交：

```bash
git commit -m "<提交信息>"
```

#### 步骤4: 询问是否推送

提交成功后：

```text
提交成功！是否推送到远程仓库？
1. 是，推送到 origin
2. 是，推送到其他远程仓库
3. 否，稍后手动推送
```

### 6. 推送流程

#### 检查远程仓库

```bash
git remote -v
```

#### 单个远程仓库

如果只有一个远程仓库（通常是 origin）：

```bash
git push origin <当前分支>
```

#### 多个远程仓库

如果有多个远程仓库，询问推送到哪个：

```text
检测到多个远程仓库，请选择推送到哪个：
1. origin (默认)
2. upstream
3. 其他...
```

然后执行推送：

```bash
git push <选择的远程仓库> <当前分支>
```

### 7. 用户提前说明直接推送

如果用户在请求时明确说明"直接推送"或类似指令，则跳过询问步骤，直接执行：

```bash
git commit -m "<生成的提交信息>" && git push origin <当前分支>
```

## 特殊情况处理

### 1. 没有暂存文件

如果没有文件被暂存：

```text
当前没有暂存的文件。请先使用 `git add` 添加要提交的文件。
```

### 2. 有未暂存的变更

如果工作区有变更但未暂存：

```text
检测到未暂存的变更。是否要暂存所有变更并提交？
1. 是，暂存所有变更并提交
2. 否，我只想提交已暂存的文件
3. 显示详细变更
```

### 3. 提交信息冲突

如果提交信息不符合规范：

```text
提交信息不符合规范，请检查：
- 类型是否正确
- 范围是否使用小写kebab-case
- 主题是否使用中文且不超过50字符
```

### 4. 推送失败

如果推送失败：

```text
推送失败，可能的原因：
1. 远程仓库有新的提交，请先拉取
2. 权限不足
3. 网络问题

是否尝试先拉取再推送？
1. 是，执行 git pull --rebase 然后推送
2. 否，我手动处理
```

## 回答风格要求

- 全程使用中文说明
- 提交信息必须符合行业规范
- 代码示例中的变量名、命令保持英文
- 流程清晰，步骤明确
- 主动询问用户意图，避免假设

## 质量要求

### 提交信息质量

- 类型必须准确反映变更性质
- 范围必须准确反映影响模块
- 主题必须简洁明了，使用中文
- 正文必须解释为什么修改
- 页脚必须正确关联 Issue

### 流程完整性

- 必须检查 git 状态
- 必须分析变更内容
- 必须生成符合规范的提交信息
- 必须询问用户确认
- 必须处理推送逻辑

### 错误处理

- 必须处理各种异常情况
- 必须提供清晰的错误信息
- 必须提供解决方案建议

## 使用示例

### 基本使用

用户：帮我提交代码

助手：
1. 检查 git 状态
2. 分析变更
3. 生成提交信息
4. 询问是否提交
5. 提交后询问是否推送

### 直接推送

用户：帮我提交代码并直接推送到 origin

助手：
1. 检查 git 状态
2. 分析变更
3. 生成提交信息
4. 直接执行提交和推送

### 多远程源

用户：帮我提交代码

助手：
1. 检查 git 状态
2. 分析变更
3. 生成提交信息
4. 询问是否提交
5. 提交后检查远程仓库
6. 如果多个远程源，询问推送到哪个
7. 执行推送
